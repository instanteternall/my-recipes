# 我的菜谱项目开发规范文档

## 1. 项目概述

项目名称：我的菜谱
项目类型：前后端分离应用
技术栈：
- 前端：Vue 3 + Vite + TypeScript + Element Plus
- 后端：Spring Boot 3 + Java 17 + MyBatis-Plus
- 数据库：MySQL 8.0

## 2. 目录结构规范

```
frontend/
```

## 3. 命名规范

### 3.1 前端命名规范

- **组件名**：使用 PascalCase，如 `RecipeCard.vue`
- **变量名**：使用 camelCase，如 `recipeList`
- **常量名**：使用 UPPER_SNAKE_CASE，如 `MAX_RECIPES`
- **方法名**：使用 camelCase，如 `getRecipeById`
- **文件名**：组件文件使用 PascalCase，其他文件使用 kebab-case，如 `recipe-detail.vue`

### 3.2 后端命名规范

- **类名**：使用 PascalCase，如 `RecipeController`
- **方法名**：使用 camelCase，如 `getRecipeById`
- **变量名**：使用 camelCase，如 `recipeService`
- **常量名**：使用 UPPER_SNAKE_CASE，如 `MAX_PAGE_SIZE`
- **包名**：使用小写字母，如 `com.recipes.controller`
- **数据库表名**：使用 snake_case，如 `recipe`
- **数据库字段名**：使用 snake_case，如 `recipe_name`

## 4. 代码规范

### 4.1 前端代码规范

- 使用 TypeScript 进行类型定义
- 使用 ESLint 和 Prettier 进行代码风格检查
- 组件使用组合式 API
- 代码注释使用 JSDoc 格式
- 缩进使用 2 个空格
- 字符串使用单引号
- 大括号使用 K&R 风格

### 4.2 后端代码规范

- 使用 Java 17 语法
- 代码注释使用 Javadoc 格式
- 缩进使用 4 个空格
- 大括号使用 K&R 风格
- 方法参数超过 3 个时使用 builder 模式
- 异常处理使用统一的全局异常处理器

## 5. 数据库规范

- 使用 InnoDB 存储引擎
- 表结构设计遵循第三范式
- 主键使用自增 ID
- 外键使用适当的索引
- 字段类型选择合理，避免过度使用 VARCHAR
- 时间字段使用 DATETIME 类型

## 6. API 规范

### 6.1 接口命名规范

- 使用 RESTful 风格
- 路径使用小写字母，单词间用连字符连接
- GET 请求用于获取资源
- POST 请求用于创建资源
- PUT 请求用于更新资源
- DELETE 请求用于删除资源

### 6.2 响应格式规范

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

- code: 状态码，200 表示成功，其他表示失败
- message: 响应消息
- data: 响应数据

## 7. 版本控制规范

- 使用 Git 进行版本控制
- 分支命名规范：
  - feature/xxx：新功能开发
  - bugfix/xxx：bug 修复
  - hotfix/xxx：紧急修复
- 提交信息规范：
  - feat: 添加新功能

## 8. 部署规范

- 前端使用 Vite 构建，部署到静态资源服务器

- 环境配置：
  - dev：开发环境

  - prod：生产环境

## 12. 主动测试（Proactive Testing）规范

### 12.1 定义

**主动测试**是指开发人员在进行任何代码修改后，主动执行的测试活动，以确保修改不会引入新的问题或破坏现有功能。

### 12.2 需要测试的修改内容

以下修改内容必须执行主动测试：

- **代码修改**：
  - 新增功能代码
  - 修改现有功能代码
  - 修复 Bug
  - 代码重构
  - 性能优化
  - 安全修复

- **配置修改**：
  - 环境配置变更
  - 依赖版本更新
  - 构建配置修改

- **API 变更**：
  - 接口参数变更
  - 响应格式变更
  - 业务逻辑变更

### 12.3 测试要求

#### 12.3.1 前端测试

- **单元测试**：
  - 组件功能测试
  - 工具函数测试
  - 状态管理测试
  - 路由测试

- **集成测试**：
  - 页面集成测试
  - API 调用测试
  - 表单提交测试

- **回归测试**：
  - 确保现有功能正常
  - 确保响应式布局正常
  - 确保用户交互正常

#### 12.3.2 后端测试

- **单元测试**：
  - 服务层方法测试
  - 工具类测试
  - 业务逻辑测试

- **集成测试**：
  - 控制器测试
  - 数据库操作测试
  - API 接口测试

- **回归测试**：
  - 确保现有接口正常
  - 确保数据库操作正常
  - 确保权限控制正常

### 12.4 测试步骤

#### 12.4.1 前端测试步骤

1. **准备测试环境**：
   - 启动开发服务器
   - 确保测试依赖已安装

2. **执行单元测试**：
   - 运行 `npm test` 执行单元测试
   - 确保所有测试通过

3. **执行集成测试**：
   - 手动测试修改的功能
   - 验证表单提交
   - 验证 API 调用

4. **执行回归测试**：
   - 测试相关功能页面
   - 验证响应式布局
   - 验证用户交互流程

5. **验证构建**：
   - 运行 `npm run build` 确保构建成功

#### 12.4.2 后端测试步骤

1. **准备测试环境**：
   - 启动数据库
   - 运行测试数据脚本

2. **执行单元测试**：
   - 运行 `mvn test` 执行单元测试
   - 确保所有测试通过

3. **执行集成测试**：
   - 测试 API 接口
   - 验证数据库操作
   - 测试业务逻辑

4. **执行回归测试**：
   - 测试现有接口
   - 验证权限控制
   - 测试异常处理

5. **验证构建**：
   - 运行 `mvn clean package` 确保构建成功

6. **服务重启验证**（后端）：
   - 如果改动需要重启后端服务才会生效，在测试通过后主动重启服务
   - 确保所有改动在重启后正确生效

### 12.5 成功标准

测试完成的成功标准：

- **单元测试**：100% 通过
- **集成测试**：所有功能正常运行
- **回归测试**：现有功能无回归问题
- **构建验证**：构建过程无错误
- **代码质量**：无 lint 错误和类型错误

### 12.6 测试结果文档

测试结果必须记录在以下位置：

- **代码提交信息**：包含测试结果摘要
- **测试报告**：生成并保存测试报告
- **问题跟踪**：如有测试失败，创建相应的 issue

测试结果文档应包含：

- 测试时间
- 测试环境
- 测试内容
- 测试结果
- 问题描述（如有）
- 修复方案（如有）

### 12.7 验证机制

为确保主动测试的执行，实施以下验证机制：

1. **代码提交前验证**：
   - 提交前运行测试脚本
   - 确保测试通过后才能提交

2. **CI/CD 验证**：
   - 在 CI 流程中执行测试
   - 测试失败则构建失败
   - 阻止未通过测试的代码合并

3. **代码审查验证**：
   - 代码审查时检查测试覆盖
   - 确保新代码有相应测试
   - 验证测试结果文档

4. **质量门禁**：
   - 设置测试覆盖率阈值
   - 设置代码质量阈值
   - 未达到阈值的代码不允许发布

### 12.8 测试工具推荐

- **前端**：
